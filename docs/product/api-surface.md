## 2. API Design Principles

The following ten principles apply to all Phase 1 API design decisions.

**1. Resource-oriented design.** Group endpoints by the resource they operate on, not by the
page or UI surface that consumes them. One resource endpoint may serve multiple pages.

**2. Explicit actor boundaries.** Every endpoint specifies its actor (anonymous / fan /
creator-approved / admin). Endpoints accessible by multiple actors are explicitly noted with
behavior differences called out.

**3. Public read vs. authenticated read are separate concerns.** Public endpoints never
accidentally return protected fields. Authenticated requests on the same path may receive
richer data (e.g., entitlement-aware content fields), but the endpoint handles both cases
explicitly, not by accident.

**4. Permission check before entitlement check.** Evaluate role and account status first (is
this actor allowed to call this endpoint?), then evaluate entitlement (does this specific fan
have access to this specific content?). Skipping the permission layer is never acceptable to
save a query.

**5. Idempotency for sensitive writes.** Checkout initiation, cancellation submission, and
onboarding submission must be safe to retry. Business-rule deduplication (fan + resource +
type uniqueness) is the primary mechanism. Idempotency-key header support is an open decision
(see Section 10, A7).

**6. Webhook-driven billing state transitions.** `MembershipSubscription` status transitions
are never driven by client API calls. They are driven exclusively by payment gateway webhook
events received at `POST /webhooks/billing`. Client APIs read resulting state; they do not
write it. Specifically: `MembershipSubscription` is created by the billing webhook
(`payment_succeeded`), not at checkout initiation. Checkout initiation creates an `Order`
record only.

**7. Admin actions are always auditable.** Every admin write operation creates a
`ModerationAction` record in the same database transaction. No admin endpoint is write-only
without an audit trail. `admin_note` from `ModerationAction` is never returned to non-admin
callers.

**8. Dual-status visibility enforced uniformly.** Content visibility checks always evaluate
both `publish_status` (creator-controlled) and `moderation_status` (admin-controlled).
`moderation_status = removed` is an absolute override â€” no entitlement or role grants access.
`CreatorProfile.status = suspended` similarly overrides all content visibility for that
creator's content.

**9. Storage keys never returned raw.** `PostAttachment.storage_key`,
`Product.file_storage_key`, and `Product.preview_file_storage_key` are internal storage
references. Download access always returns a signed, time-limited URL generated by the storage
module after an entitlement check. The key itself is never in any API response.

**10. Avoid over-fragmentation in Phase 1.** Prefer fewer, broader endpoints over a large
number of narrow ones. Defer fine-grained filtering until post-launch unless explicitly
required by a specific Phase 1 page family.

## 3. Actor Groups

| Actor | Description | Key API capabilities |
|---|---|---|
| `anonymous` | Unauthenticated visitor | Read public creator pages, home, discovery, legal docs; submit signup/login |
| `fan` | Authenticated user with `has_fan_role = true` and `account_status = active` | Checkout, library, subscriptions, purchases, notifications, cancellation |
| `creator:onboarding` | User with `CreatorProfile.status = onboarding` | Onboarding flow read/write only |
| `creator:pending` | Creator where `status = pending_review` | Read-only onboarding status; no dashboard writes |
| `creator:approved` | Creator where `status = approved` | Full dashboard: posts, plans, products, collections, overview |
| `creator:rejected` | Creator where `status = rejected` | Read rejection reason; resubmit onboarding |
| `creator:suspended` | Creator where `status = suspended` | No content writes; no fan-facing page served |
| `admin` | User where `is_admin = true` | Creator review queue; moderation; reports; suspension; all-read |
| `system/webhook` | Internal trusted caller (gateway-signed) | `POST /webhooks/billing` only; verified by signature, not JWT |

**Dual-role note:** A user may be both `fan` and a creator in any creator state simultaneously
(data-model.md Assumption A1). Fan endpoints remain accessible to any creator account
regardless of creator status. The creator state affects only creator-specific endpoints.

**Email verification gate:** `fan` entitlement to checkout and gated content access requires
`User.email_verified_at IS NOT NULL` (permission-matrix.md; data-model.md Â§6.2). Endpoints
that enforce this return `403 UNVERIFIED_EMAIL` if the gate is not met.

### Family 1 â€” Auth

**Purpose:** Account creation, login, session management, password reset, and email
verification. Serves page families 3, 4, 5, and 6.

**Primary actors:** `anonymous` (registration, login, reset); any authenticated user (logout,
resend verification, get-current-user)

**Main resources:** User, OAuthAccount, PasswordResetToken, EmailVerificationToken,
ConsentRecord, LegalDocumentVersion

**Core operations:** register-fan, register-creator, login, logout, google-oauth-exchange,
get-current-user, request-password-reset, confirm-password-reset, verify-email,
resend-verification-email

**Permission dependencies:** All registration and login endpoints are public. `get-current-user`
and `logout` require a valid session. `resend-verification` requires a valid session.
`account_status = suspended` blocks login.

**Entitlement dependencies:** None directly. `email_verified_at` set here gates checkout and
gated content access elsewhere.

**Billing dependencies:** `BillingCustomer` is not created at registration â€” only on first
checkout.

**Moderation dependencies:** Suspended users attempting login receive `403 ACCOUNT_SUSPENDED`
with support contact information.

**Important side effects:**
- Fan registration: `ConsentRecord` created for `terms_of_service` + `privacy_policy` at the
  current `LegalDocumentVersion`. `EmailVerificationToken` created. Welcome email sent.
- Creator registration: same as fan plus `CreatorProfile` created (`status = onboarding`,
  `onboarding_last_step = 0`). `ConsentRecord` created for same document types.
- Password reset confirm: all existing sessions for the user are invalidated.
- Email verification: `User.email_verified_at` set.

**Audit / logging:** No `ModerationAction`. `ConsentRecord` is the audit trail for consent.

**Deferred:** Two-factor authentication; account email-change flow; account deletion; account
settings management.

### Family 14 â€” Checkout

**Purpose:** Initiate and track membership, product, and collection checkout. Serves page
families 16, 17, and 18.

**Primary actors:** `fan` (email-verified)

**Main resources:** Order, Product, Collection, MembershipPlan, BillingCustomer,

**Permission dependencies:** `fan` only. `User.email_verified_at IS NOT NULL` required â€”

**Entitlement dependencies:**
- Duplicate purchase guard on product/collection checkout: if a `ProductPurchase` or

- Fan cannot initiate a new subscription to a tier while a non-expired subscription to that

- `BillingCustomer` created on first checkout if not already existing.
- Checkout initiation: `Order` created (`status = pending`) only. Gateway payment intent

- Payment confirmation is asynchronous (webhook-driven). The billing webhook

**Important side effects:**
- `initiate-membership-checkout`: Creates `Order` (pending) only.

**Audit / logging:** `Order` and `Invoice` records are the billing audit trail.

## 6. Cross-Cutting Rules

### 6.1 Authentication and Session

**Resolved (OD-20, OD-22 â€” 2026-03-27):** JWT bearer token with refresh token pair.

- All authenticated routes require a valid, non-expired `Authorization: Bearer <access_token>` header.
- Login (`POST /auth/login`) returns `{ access_token, refresh_token, expires_in }`.
- Access tokens are short-lived. Refresh tokens are renewed via `POST /auth/refresh`.
- Logout revokes the refresh token for the requesting session.
- Password reset invalidates all active refresh tokens for the user.
- Admin login (`POST /admin/auth/login`) issues a token with a `scope: admin` claim.
  Fan-issued tokens from `POST /auth/login` cannot satisfy admin route guards even if
  the user has `is_admin = true`.
- Admin routes require both a valid session and `User.is_admin = true`.
- Redis-backed HttpOnly session cookies are not used in Phase 1.

### 6.2 Authorization Layering

Checks applied in strict order for every request:

1. **Role check** â€” is the route accessible by this actor type? (`is_admin`, `has_fan_role`,
   `CreatorProfile.status`)
2. **Account status** â€” is the actor's account active? (`User.account_status = active`)
3. **Email verification** â€” is the actor's email verified? (required for checkout and gated
   content access: `User.email_verified_at IS NOT NULL`)
4. **Ownership** â€” does the actor own the resource? (`fan_user_id`, `creator_profile_id`
   match)
5. **Entitlement** â€” does the actor have content entitlement? (subscription/purchase check for
   fan-gated reads)
6. **Dual-status visibility** â€” does the content pass compound visibility? (`publish_status` +
   `moderation_status`)

No layer may be skipped.

### 6.3 Entitlement Enforcement

Content entitlement is evaluated at request time from live entity state. It is never stored as
a derived or cached field. The entitlement service reads `MembershipSubscription.status`,
`current_period_end`, `grace_period_ends_at`, `PostPurchase.access_revoked_at`,
`ProductPurchase.access_revoked_at`, and `CollectionPurchase.access_revoked_at` at each
relevant request. (data-model.md Â§6)

### 6.4 Pagination Defaults

All list endpoints return paginated responses. Default page size: **20**. Maximum page size:
**100**. Pagination strategy (cursor-based or offset-based) is an implementation detail. All
list responses include `total`, `page`, `per_page`, and `has_more` fields (or equivalent
cursor metadata).

### 6.5 Filtering

Filters are documented per endpoint family (Section 4) and inventory (Section 5). Basic enum
and date-range filters are in scope for Phase 1. Full-text content search is deferred. Creator
discovery supports `category` enum filter and `q` (display_name match) at minimum.

### 6.6 Idempotency

Checkout initiation, cancellation submission, and onboarding submission include business-rule
deduplication. At minimum: re-submitting the same checkout within a deduplication window
returns the existing pending `Order` rather than creating a new one. Idempotency-key header
support is an open decision (Section 10, A7).

### 6.7 Webhook Signature Verification

`POST /webhooks/billing` validates the gateway's request signature before processing any
event. Requests without a valid signature return 401 immediately. Webhook handlers must be
idempotent by `event_id` â€” the same event may be delivered multiple times by the gateway.

### 6.8 Error Response Shape

All error responses follow a consistent shape:

```
{
  "statusCode": <HTTP status>,
  "error": "<HTTP error name>",
  "code": "<machine-readable error code>",
  "message": "<human-readable description>"
}
```

Key error codes in Phase 1:

| Code | HTTP Status | Meaning |

| `ACCOUNT_SUSPENDED` | 403 | Account suspended; contact support |
| `UNVERIFIED_EMAIL` | 403 | Email not verified; checkout and gated access blocked |

### 6.9 Auditability

All admin write operations create a `ModerationAction` record in the same database transaction
as the primary write. If `ModerationAction` creation fails, the primary write rolls back.
`admin_note` on `ModerationAction` is never returned in any API response to non-admin callers,
regardless of endpoint or query parameter.

## 7. Billing and Checkout Surface

### 7.1 Synchronous vs. Asynchronous Operations

**Synchronous (inline API response):**
- Order creation (`POST /checkout/membership`, `/checkout/product`, `/checkout/collection`)
- Order status read (`GET /checkout/orders/:order-id`)
- Cancellation submission (`POST /account/subscriptions/:sub-id/cancel`)
- Invoice and subscription list reads

**Asynchronous (webhook-driven via `POST /webhooks/billing`):**
- Payment confirmation â†’ `Order.status â†’ completed`
- **Membership activation** â†’ `MembershipSubscription` created with `status = active`
- Payment failure at initial checkout â†’ `Order.status â†’ failed` (no `MembershipSubscription`)
- Renewal success â†’ `Invoice` created; period dates updated; fan notified
- Renewal failure â†’ `MembershipSubscription.status â†’ grace_period`; `grace_period_ends_at`
  set; fan notified
- Grace period resolution (payment updated + retry success) â†’ `status â†’ active`; Invoice
  created
- Grace period expiry â†’ `status â†’ expired`; fan notified
- Subscription expiry (cancelled period end) â†’ automated transition via scheduled job, not
  webhook

### 7.2 Membership Checkout Flow

1. Fan: `POST /checkout/membership` â†’ `{creator_username, plan_id, billing_interval}`
   - `Order` (pending) created **only**
   - Response: `{order_id, payment_intent, gateway_form_config}`
2. Fan completes payment in gateway UI (hosted iframe or hosted fields)
3. Gateway: `POST /webhooks/billing` (event: `payment_succeeded`)
   - `Order â†’ completed`
   - `MembershipSubscription` created with `status = active` (subscription begins here)
   - `Invoice` (subscription_charge) created
   - `Notification` (`order_confirmed`) sent to fan
4. Fan: `GET /checkout/orders/:order-id` â†’ `order.status = completed`
   - Frontend renders `/checkout/onay/[order-id]` using order data

**Initial payment failure:** Gateway sends `payment_failed` webhook â†’ `Order â†’ failed`. No
`MembershipSubscription` record exists. Fan sees `/checkout/hata/[order-id]` page via
`GET /checkout/orders/:order-id` (order.status = failed).

### 7.3 Product and Collection Checkout Flow

Same asynchronous pattern as Â§7.2 but:
- No `MembershipSubscription` involved
- On `payment_succeeded` webhook: `ProductPurchase` or `CollectionPurchase` created; `Invoice`
  created; `Notification` (`order_confirmed`) sent
- On `payment_failed` webhook: `Order â†’ failed`; no purchase record created

### 7.4 Cancellation Interaction with Billing

- Cancellation is client-initiated via `POST /account/subscriptions/:sub-id/cancel`
- Sets `cancelled_at`; `status` remains `active`
- Access continues until `current_period_end` (no immediate webhook to gateway required;
  billing module flags that next renewal should not be attempted)
- When `current_period_end` passes â†’ `MembershipSubscription.status â†’ expired` via scheduled
  job, not an API call

### 7.5 Grace Period Flow

1. Renewal charge fails â†’ gateway sends `payment_failed` webhook for renewal event
2. `MembershipSubscription.status â†’ grace_period`; `grace_period_ends_at` set
3. Fan receives `Notification` (`membership_renewal_failed`) with payment update link
4. Fan updates payment method via `PUT /account/billing/payment-method`
5. Platform or gateway retries charge â†’ `payment_succeeded` â†’ `status â†’ active`; Invoice
   created
6. If `grace_period_ends_at` passes without resolution â†’ `status â†’ expired`; fan notified

### 7.6 Payment Method Reconciliation

`GET /checkout/orders/:order-id` is the fan-facing reconciliation endpoint. Confirmation and
error pages poll this endpoint to determine final order state. Polling with exponential
backoff is the MVP pattern. Server-sent events / real-time updates are deferred (Section 10,
A4).

### 7.7 Platform Fee

Platform fee percentage is a configuration value, not stored in the data model. Revenue
figures on the dashboard overview and invoice aggregates reflect amounts according to billing
module configuration (pre-fee vs. post-fee â€” see Section 10, A6).

## 8. Moderation and Trust Surface

### 8.1 What Admin Can See That Normal Users Cannot

- `admin_note` on any `ModerationAction` â€” never returned to non-admin callers

- `CreatorApplication.rejection_reason` â€” visible in application detail to admin

### 8.2 Content Restriction Visibility Matrix

|---|---|---|---|

All `/admin/*` endpoints require `User.is_admin = true`. Any non-admin request returns 403
